# Wave Plan — Daily Driver
_2026-06-10 · PRD for sign-off before any build work. Goal chosen at gate: **Kettlecal running on James's phone as the daily training tool**, tight scope, 1–2 weeks of sessions._

## What already exists (verified in tree today — not re-proposed)

Since the 06-09 review, three commits landed: dead-code + hold-edge engine fixes with the doc refresh (c12b6f8), **active-session persistence + resume prompt with tests** (5c96021), and **CI: tsc + vitest + web export + Pages deploy** (8070b76). Catalog v3 (this morning's audit work: overhead restriction + migration, per-emphasis focus days, tier field, 11 new exercises, 116 tests) is implemented and green but **uncommitted**, alongside the visual restyle. The only structural gap left between here and a phone is distribution: no `eas.json`, no bundle identifiers, and MMKV v4 rules out Expo Go.

## Problem

The app is feature-complete enough to train with daily, but it runs nowhere except a web export and a Metro-tethered simulator. Until it's on a phone surviving real kitchen-schedule training — phone locked mid-set, killed mid-workout, no network — the engine's quality is theoretical and the tuning knobs (emphasis priorities, rest times, progression windows) can't be tuned, because tuning "by feel" requires feel.

## Success criteria (wave is done when all are true)

1. Zero uncommitted work; CI green on `main`; GitHub Pages serving the current build.
2. Kettlecal installed on James's phone as a **standalone build** (no Metro tether) and used for real sessions.
3. On-device shakedown passed: force-kill mid-workout → relaunch → resume prompt restores logged sets; backup export/import round-trips; EMOM beep and haptics fire; **rest timer behaviour with the screen locked is verified** (JS timers pause in background — this is the most likely real-world failure and has never been tested on hardware).
4. ≥5 real training sessions logged by wave end, with tuning notes captured in `docs/`.
5. An explicit end-of-wave decision: does `autoAdjust` get switched on? (That decision gates the deferred T3 narration work — building narration before deciding is waste.)

## Scope

**In:** committing current work · EAS setup (identifiers, `eas.json` with `development` + `preview` profiles — preview embeds the JS bundle so it runs untethered; that's the daily-driver build) · install on device · scripted on-device shakedown · field-use tuning pass.

**Out (explicitly deferred, do not pull in):** T3 pain-narration + undo (gated on the autoAdjust decision) · recovery-based rotation · readiness check-in · routine templates · trend charts · wearables · expo-av → expo-audio migration (only touch if the device beep breaks) · F7 double-KB quantity · component test suite (one exception below).

## Plan

**S1 — Hygiene (~1 hour).** Two commits: `ui:` restyle + SynthCard, then `feat(catalog):` v3 + audit docs. Push; watch CI complete; confirm Pages source is configured (one-time repo setting — the workflow can't set it). Everything after this is protected by CI.

**S2 — Distribution (Android-only; decided 2026-06-10).** Web ships automatically with S1 (Pages deploy already in CI) and serves as the interim/home-screen version — but it is NOT the daily driver: the web build falls back to browser storage, which the browser may evict, reopening the data-loss hole the resume-workout work just closed. The daily driver is an **EAS preview APK**: add `android.package`; create `eas.json` with `development` + `preview` (APK) profiles; one cloud build; sideload. No developer-account cost or lead time. Budget for first-native-build fallout (MMKV v4, Reanimated 4, expo-av on SDK 56) — historically this always costs half a session.

**S3 — Shakedown (on device, scripted checklist).** Kill-mid-workout/resume; backup round-trip; locked-screen rest timer; EMOM beep + haptics; a full fake session end-to-end; day-turnover regeneration next morning. Fix what breaks; anything non-blocking goes to the deferred list, not into the wave.

**S4–S5 — Field use + tuning.** Train with it. Capture friction notes per session (one line each is enough). End of wave: tune emphasis priority lists and rest seconds from real feel; review logged pain/RPE; make the autoAdjust on/off call; write the next-wave seed list from the friction notes.

**Scope exception:** if S3/S4 force any change to `SetLogger`, add the single interaction test (log a set → store contains it) in the same commit. It guards the one component where training data is born.

## Constraints & risks

Android APK = free sideload, zero lead time (iOS dropped from this wave — decided 2026-06-10). The locked-screen timer risk is real and unknown until hardware testing; if it fails, the fix (expo-notifications scheduled alert or background-aware timestamps) is small but must come from measurement, not guesswork. Web-build caveat: browser storage is evictable — fine for a demo, not for training history; the APK is where real sessions get logged. CI runs on fresh Ubuntu installs so the platform-binary issue from my sandbox doesn't apply there.
